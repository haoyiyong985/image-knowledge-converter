## 重要教训：恢复图片的正确查找方法

### 问题背景
2026-04-10 用户想要恢复4月9日处理的图片进行重新测试。

### 踩坑过程
1. 最初查看日志，误以为只有4张图片
2. 用户说是V6处理的约11张
3. 检查历史学文件夹发现7张
4. 用户纠正还有其他分类的错误分类
5. 最终发现：4月9日处理的图片分散在**5个分类**，共**12张**

### 根本原因
**分类错误**：V4/V6版本的分类器将历史学内容错误地分配到了其他分类（生活方式、营养科普、育儿教育、中医养生）。

### 正确的查找方法
```powershell
# 按创建时间查找特定日期处理的图片（最准确）
Get-ChildItem "d:\新建文件夹\已处理图片" -Recurse -File | 
  Where-Object { $_.CreationTime -ge "2026/04/09" -and $_.CreationTime -lt "2026/04/10" } |
  ForEach-Object { $_.Directory.Name + "\" + $_.Name }
```

### 经验总结
1. **日志不够准确**：不能完全依赖日志文件判断处理了哪些图片
2. **用文件创建时间判断**：通过 `CreationTime` 属性可以准确判断图片的处理时间
3. **检查所有分类**：图片可能被错误分类到其他文件夹，需要搜索所有已处理分类
4. **分类错误会导致图片分散**：历史内容被分散到5个分类，需要全部找回

## 2026-04-15 V9.2 修复

### JSON截断根因
- LLM在content字段中使用未转义双引号（如"阉党"），破坏JSON结构
- V9.1将content放在JSON里，content越长包含引号概率越高，JSON越容易坏
- 降级提取只拿到title/category/summary，content丢失→内容重写功能完全失效

### V9.2 修复方案
1. **分次调用LLM**：第1次获取title/category/summary（短JSON），第2次整理content（纯Markdown）
2. **核心实体匹配**：从标题开头提取2-3字核心实体（人名/地名），相同则返回0.80相似度→解决钱谦益文档不合并问题
