## V9.1 LLM内容重写升级（20:50）

### 问题根因
用户测试V9.0发现文档内容没有被整理：OCR原始文本（含界面残片如"豆包>"、时间栏）直接写入文档，内容质量差。

根本原因：V9.0的LLM调用只做了命名/分类/摘要，**内容本身从未经过LLM整理**。

### 修复：V9.1 LLM内容重写

| 改动 | 说明 |
|------|------|
| prompt扩展 | 新增`content`字段，要求LLM去掉APP界面残片、修正OCR错误、用##/###格式化章节 |
| 传入文本扩大 | 500字 → 1200字（更完整的上下文） |
| final_content | 优先用`llm_content`，LLM失败则退回`clean_text`（OCR去噪文本） |
| JSON解析改进 | DOTALL支持跨行content字段；支持代码块包裹（```json...```） |

### 现在的处理流程
```
OCR原文
  → ContentOrganizer（去噪/OCR修复）→ clean_text
  → LLMAnalyzer.analyze()
      → prompt: 返回 {title, category, summary, content}
      → content = 整理后的Markdown正文
  → final_content = llm_content（有）或 clean_text（无）
  → 文档生成（使用final_content）
```
