# V9 LLM全覆盖方案：用混元Lite彻底替代正则

## 一、V2 vs V8.3 根本差异分析

### V2为什么没有这些问题？

| 环节 | V2 的做法 | V8.3 的做法 | 为什么V2没问题 |
|------|----------|------------|---------------|
| **文档命名** | `extract_title()` 简单提取前10行中第一个有意义的短句 | 复杂正则+置信度系统：人名/历史/饮食等10+模式，200+行正则代码 | V2**不追求精确命名**，提取到什么就用什么，虽然不够好但**至少不会错误** |
| **内容整理** | `structure_content()` 只对**语义关键词**（功效/做法/禁忌等）添加 `###` | 复杂模式匹配：数字序号→`##`、多列排版重建、编号格式降级 | V2的规则**很少触发**（只匹配特定关键词），所以Markdown标记残留少 |
| **文档合并** | `find_similar_doc()` 用字符集 Jaccard 相似度 > 0.6 | normalize后bigram Jaccard + 朝代统一 + 包含关系 + 内容重叠 | V2的相似度**足够宽松**（字符集级别），而V8的正则命名产生错误标题后，相似度匹配反而失败 |
| **分类** | `THEME_KEYWORDS` 关键词匹配 + `ClassifierEngine` | `SmartClassifier` 关键词匹配 + 置信度 | 本质相同，V2更简单直接 |

### V8系列的"过度工程"问题

V2 → V6 → V7 → V8 的演进过程中，每发现一个问题就加一条正则：
- `一张图看完` → 加正则 → 置信度过高 → 又要调
- `苏无锡人` → 加地名黑名单 → OCR换种断法又绕过 → 再加
- `明朝(上)(下)` → normalize去后缀 → 但命名是错的还是匹配不上

**正则系统变成了一个不断打补丁的"怪兽"**，每修一个case引入另一个edge case。

## 二、V9 核心设计：LLM全覆盖

### 设计原则

> **正则只做过滤（排除明显垃圾），LLM做所有决策**

混元Lite是**免费+无限调用+2秒响应**的。每张图片调1-2次LLM完全可行。

### 三大决策全部交给LLM

```
图片 → OCR → 原始文本
              ↓
         ┌────────────────────┐
         │  混元Lite 一次性调用  │
         │                    │
         │  返回：             │
         │  1. title: 文档标题  │
         │  2. category: 分类  │
         │  3. summary: 内容摘要│
         └────────────────────┘
              ↓
         正则只做：去噪、格式化
```

### LLM Prompt 设计

```
你是一个图片内容分析助手。以下是从图片中OCR识别的原始文本（可能包含识别错误）。
请分析内容，返回JSON格式：

{
  "title": "简洁的文档标题（2-15个字，不要包含'一张图看完'等平台水印）",
  "category": "从以下分类中选择：历史文化、营养健康、生活方式、教育育儿、旅游攻略、综合知识、其他",
  "summary": "一句话概括内容（20-50字）"
}

原始文本：
{ocr_text}
```

### 需要保留的正则（仅4个，全部是过滤级）

| 正则 | 作用 | 不再用决策 |
|------|------|-----------|
| 长度<10的行 | 过滤OCR噪点 | ~~分类~~ |
| 纯符号行 | 过滤分隔符 | ~~命名~~ |
| 混合乱码行 | 过滤 `甘o亚a` | ~~标题识别~~ |
| 空行合并 | 格式化 | ~~结构化~~ |

### 内容整理：保留V2的轻量级结构化

V2的 `structure_content()` **保留但简化**：
- 只识别语义标题（功效/做法/禁忌/步骤等明确关键词）→ `###`
- 不再识别数字序号（`1.` `2.`）→ 不转 `##`
- 不再多列排版重建 → **交给LLM的summary代替**

### 文档合并：保留V8.3的相似度系统

V8.3的合并逻辑（normalize + 朝代统一 + 包含关系）**本身是正确的**，问题在于命名错误导致相似度计算失败。LLM命名后这个问题自然消失。

## 四、LLM调用成本分析

| 场景 | V8.3 调用次数 | V9 调用次数 |
|------|-------------|------------|
| 11张图片 | 3次（低置信触发） | **11次**（每张1次） |
| 100张图片 | ~30次 | **100次** |
| 混元Lite成本 | **免费** | **免费** |
| 额外耗时 | ~6秒 | ~22秒 |
| 总耗时(100张) | ~10分钟 | ~12分钟 |

**结论：每张图片多花0.2秒，换来100%的命名准确性。完全值得。**

## 五、V2保留的优秀设计

V9应该从V2继承（而不是从V8继承）的设计：

1. ✅ **简洁的 `extract_title()`** 作为LLM失败时的兜底
2. ✅ **V2的 `structure_content()`** 轻量级结构化（只识别语义关键词）
3. ✅ **V2的 `ContentOrganizer.organize()`** 流程：提取来源→清理→结构化
4. ✅ **V2的 `generate_word()`** Word生成逻辑（读合并MD→生成）
5. ✅ **V2的IMA同步** 逻辑不变

## 六、风险与兜底

| 风险 | 兜底方案 |
|------|---------|
| 混元Lite不可用 | 检测超时/错误→降级到V2的简单提取 |
| LLM返回格式错误 | JSON解析失败→正则兜底 |
| LLM返回低质量标题 | 长度检查（<2字或>20字）→正则兜底 |
| LLM分类不在预设列表 | 归入"其他" |
